Lesson 2 - Part 2

Create SSH Key Pair

Part 2: Create SSH Key Pair (15 minutes)

üìò What is SSH and how do key pairs work?

What it is: SSH (Secure Shell) is a way to remotely access and control a computer over a network. A key pair consists of two cryptographic keys: a private key (stays on your computer) and a public key (goes on the server). Think of it like a lock and key - but way more secure than a password.

Why we need it: Passwords can be guessed, leaked, or brute-forced. SSH keys use mathematical encryption that's nearly impossible to crack. The server has the public key (lock), you have the private key. When you connect, the server challenges you to prove you have the matching private key, all without ever transmitting the key itself over the network.

Key details: AWS doesn't support password authentication for EC2 instances - you must use SSH keys. When you create a key pair, AWS generates the pair and gives you the private key exactly once. If you lose it, you can't recover it - you'd need to create a new key pair. The public key is stored on the EC2 instance in ~/.ssh/authorized_keys.

Common gotcha: Never share your private key or commit it to version control. If someone gets your private key, they can access all servers that trust it. Also, SSH keys are region-independent but must be created per-region in AWS - a key pair created in us-east-1 won't appear in us-west-2's dropdown menus.

Step 2: Generate Key Pair

EC2
# Create key pair
aws ec2 create-key-pair \
  --key-name my-aws-key \
  --query 'KeyMaterial' \
  --output text > ~/.ssh/my-aws-key.pem
üìò What is a .pem file?

What it is: PEM (Privacy Enhanced Mail) is a file format for storing cryptographic keys and certificates. Despite the name, it's not just for email - it's a standard format for encoding keys in base64 with header/footer markers. Your .pem file contains your SSH private key in this format.

Why we need it: Cryptographic keys are binary data that can't be easily displayed or copied as text. The PEM format converts them to ASCII text that's safe to store in files, transmit, and work with across different systems. It's a widely supported standard that works with SSH, SSL/TLS, and many other security tools.

Key details: A .pem file starts with "-----BEGIN RSA PRIVATE KEY-----" and ends with "-----END RSA PRIVATE KEY-----" (or similar). Everything in between is base64-encoded binary data. The .pem extension is just a convention - it's still a plain text file. AWS EC2 generates keys in PEM format by default because it's universally compatible.

Common gotcha: PEM files must have Unix-style line endings (LF, not CRLF). If you edit them on Windows and accidentally save with Windows line endings, SSH will reject them with cryptic errors. Also, extra whitespace or characters before/after the header/footer markers will break the file. Never edit .pem files manually unless you know what you're doing.

# Secure the key
chmod 400 ~/.ssh/my-aws-key.pem
üìò Why chmod 400? What does it mean?

What it is: The chmod command changes file permissions on Unix-like systems. The number 400 means "owner can read only" - no write, no execute, and no access for anyone else. It's the most restrictive permission that still lets you use the file.

Why we need it: SSH refuses to use private keys that are too permissive - it's a security feature to prevent accidentally exposing your keys. If others can read your private key, they could copy it and impersonate you. If it's writable, malware could modify it. SSH requires private keys to be readable only by you, and chmod 400 enforces exactly that.

Key details: In Unix permissions, 4=read, 2=write, 1=execute. The three digits represent owner, group, and others. So 400 means: owner has read (4), group has nothing (0), others have nothing (0). You could also use 600 (owner read+write) which SSH also accepts, but 400 is more secure since you don't need to modify the key.

Common gotcha: Windows doesn't use Unix permissions, so chmod doesn't exist there. If you're on Windows using Git Bash or WSL, chmod works, but Windows's file properties override it. Instead, right-click the file, go to Properties > Security, and ensure only your user has access. Also, if you copy the .pem file, the copy might have different permissions - always re-run chmod 400.

EC2
# Verify it exists
aws ec2 describe-key-pairs --key-name my-aws-key

Don't lose this key! It's your only way to SSH into your servers.
