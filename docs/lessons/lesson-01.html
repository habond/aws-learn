Lesson 1 - Part 1

Your First Internet Empire

What You're Building

Deploy your own corner of the internet: a lightning-fast static website with a custom domain, served globally via CDN. You'll build something ridiculous and fun (because why not?) while learning the fundamentals of AWS.

AWS Services Used in This Lesson

S3 Object storage and static website hosting
CloudFront Content Delivery Network for global distribution
Route 53 DNS and domain management
IAM Identity and Access Management
AWS CLI Command-line interface for AWS
CloudWatch Monitoring and logging

What You'll Learn

  • S3: Object storage and static website hosting
  • CloudFront: Content Delivery Network (CDN)
  • Route 53: DNS and domain management
  • IAM: Identity and Access Management
  • AWS CLI: Command-line superpowers
  • CloudWatch: Basic monitoring
üìò What is S3 (Simple Storage Service)?

What it is: S3 is object storage - a place to store files (objects) in containers called buckets. Think of it like Dropbox or Google Drive, but designed for applications rather than humans.

Why we need it: Traditional file systems are tied to specific servers. If the server dies, your files are gone. S3 stores files redundantly across multiple data centers, making them virtually indestructible and accessible from anywhere on the internet.

Key details: Each file gets a unique URL. You pay only for what you store and transfer (pennies per GB). S3 has 99.999999999% durability - if you upload 10 million files, you might lose one every 10,000 years.

Common gotcha: S3 is not a file system - you can't "navigate folders" like on your computer. What looks like folders are just file names with slashes (like photos/vacation.jpg). This matters when you write code to access files.

üìò What is CloudFront (CDN)?

What it is: CloudFront is a Content Delivery Network - a global network of servers (called edge locations) that cache copies of your files close to your users. Amazon has 400+ edge locations worldwide.

Why we need it: If your files are stored in Virginia but your user is in Tokyo, every request travels 11,000 km round-trip (taking 200-500ms). CloudFront caches files in Tokyo, so the same request takes 5-20ms - that's 10-100x faster!

Key details: First user in a region gets files from origin (S3), then CloudFront caches them locally. Subsequent users get cached copies instantly. You control cache duration (TTL) - images might cache for days, HTML for minutes.

Common gotcha: When you update your website, old versions stay cached until expiration or you manually invalidate them. This is why your changes might not appear immediately - CloudFront is still serving the old cached version.

üìò What is Route 53 and why is it called that?

What it is: Route 53 is AWS's DNS service. DNS (Domain Name System) translates human-readable domain names like "google.com" into IP addresses like "142.250.80.46" that computers use to find servers.

Why we need it: Nobody wants to remember that your website is at 52.219.47.181. DNS lets people use yourdomain.com instead. Route 53 also handles domain registration, so you can buy and manage domains without leaving AWS.

Key details: The "53" refers to port 53, the standard port for DNS traffic - it's a nerdy inside joke. Route 53 offers different routing policies: simple (one destination), weighted (split traffic), geolocation (route based on user location), and more.

Common gotcha: DNS changes aren't instant - they need to "propagate" across the internet. This can take 5-60 minutes. During this time, some users see the old site, others see the new one. Always make DNS changes during low-traffic periods.

üìò What is IAM (Identity and Access Management)?

What it is: IAM controls who can access what in your AWS account. It manages users (people), roles (what applications can do), and policies (specific permissions like "can read S3 but not delete").

Why we need it: Your root account has unlimited power - it can delete everything and rack up unlimited bills. You never use it for daily work. Instead, create IAM users with only the permissions they need (principle of least privilege).

Key details: Users are for humans (with passwords), roles are for applications (no passwords, temporary credentials). Policies define permissions in JSON format. You can attach the same policy to many users/roles.

Common gotcha: "Access Denied" errors are usually IAM issues. If something doesn't work, check: Does your user have the right policy? Is the policy correctly written? Are you in the right region? IAM is global, but some resources are region-specific.

üìò Why use AWS CLI instead of the Console?

What it is: The AWS CLI (Command Line Interface) is a tool that lets you control AWS services by typing commands instead of clicking through the web console.

Why we need it: Clicking through the console is fine for learning, but real workflows need automation. CLI commands can be scripted, repeated, version-controlled, and shared with teammates. "Deploy the app" becomes one command instead of 50 clicks.

Key details: Every console action has a CLI equivalent (and vice versa - CLI sometimes has more options). Commands follow the pattern: aws [service] [action] [parameters]. You can output as JSON, table, or text formats.

Common gotcha: The CLI uses credentials stored on your computer (~/.aws/credentials). If commands fail with "Unable to locate credentials", run "aws configure" first. Also, some commands use "aws s3" (high-level) and others use "aws s3api" (low-level) - they're different!

Part 1: AWS Account Setup (30 minutes)

Step 1: Set Up Billing Alerts

Why: So AWS doesn't surprise you with a $500 bill

  1. Go to AWS Console ‚Üí Billing Dashboard
  2. Click "Billing preferences"
  3. Enable "Receive Billing Alerts"
  4. Set up a CloudWatch alarm for $10 threshold
  5. Add your email
üìò How do Billing Alerts work?

What it is: Billing Alerts use CloudWatch to monitor your AWS spending and send email notifications when costs exceed thresholds you define (like $10, $50, $100).

Why we need it: AWS charges for usage - every API call, every GB stored, every hour a server runs. It's easy to accidentally leave resources running or misconfigure something expensive. A $10 alert catches mistakes before they become $1,000 problems.

Key details: Billing data updates every 6-8 hours (not real-time). Set alerts lower than you think - if your threshold is $100, you might get the email when you're already at $150. You can create multiple alarms for different thresholds.

Common gotcha: Billing alerts don't stop charges - they only notify you. If you get an alert and ignore it, AWS keeps charging. You must manually stop/delete the expensive resources. Some students set $5 alert, ignore it, and end up with $200 bills.

Why: Never use root account for daily tasks

  1. Go to IAM ‚Üí Users ‚Üí Add User
  2. Username: your-name-admin
  3. Enable "AWS Management Console access"
  4. Attach policy: AdministratorAccess (we'll lock this down in Lesson 11)
  5. Important: Save credentials somewhere safe!
  6. Enable MFA (Multi-Factor Auth) - use phone authenticator app
üìò What is MFA (Multi-Factor Authentication)?

What it is: MFA requires two forms of proof to log in: something you know (password) and something you have (phone with authenticator app generating 6-digit codes that change every 30 seconds).

Why we need it: Passwords get leaked, stolen, or guessed. Even if someone gets your AWS password, they can't log in without your phone. This is critical because AWS accounts control real money - someone who hacks your account can spin up $10,000 of servers.

Key details: Use apps like Google Authenticator, Authy, or 1Password. During setup, AWS shows a QR code - scan it with your app. Save the backup codes somewhere safe (not on your phone!) in case you lose your device.

Common gotcha: If you lose your phone without backup codes, you're locked out. AWS support can help but it takes days and requires identity verification. Some people screenshot the QR code during setup - that works but is less secure if your computer gets compromised.

üìò What are AWS Regions?

What it is: AWS operates in 30+ geographic regions worldwide (like us-east-1 in Virginia, eu-west-1 in Ireland, ap-southeast-1 in Singapore). Each region is a cluster of data centers completely independent from others.

Why we need it: You choose regions based on: where your users are (latency), where data must legally reside (compliance - EU data stays in EU), and cost (some regions are cheaper). Resources in one region don't automatically exist in others.

Key details: us-east-1 (Virginia) is special - it's the oldest, cheapest, gets new features first, and is required for some global services like CloudFront SSL certificates. Most tutorials use us-east-1 as the default.

Common gotcha: You create an S3 bucket in us-east-1, then switch to eu-west-1 in the console - the bucket "disappears"! It's still there, just in the other region. Always check which region you're viewing. Some resources (like IAM) are global, others (like EC2, S3) are regional.
